2D Mapping and Navigation

2D Mapping and Navigation for Ackermann Platform

A fully operational autonomous mobile robot — sensor network, teleoperation, real-time 2D mapping, localisation, and indoor navigation with obstacle avoidance — integrated end-to-end on a ROS 2 stack and deployed on the Ackermann chassis.

This project demonstrates that a production-grade autonomous navigation pipeline can be assembled, tuned, and validated on a compact indoor platform without sacrificing architectural rigour.

System Overview

The platform is organised into three layers: sensing, middleware, and capability modules. All sensor modalities feed a single middleware layer. Three independent capability modules — teleoperation, mapping and localisation, and navigation — consume from that layer and converge on a unified Ackermann-aware motion driver.

This separation keeps each module independently testable and the overall system composable. Swapping the navigation stack, for example, does not require changes to the sensor or teleoperation layers.

  • Sensing layer — IP camera (wired); 2D LiDAR (USB); IMU (USB)
  • Capability modules — Teleoperation (desktop + mobile); 2D mapping and localisation; autonomous navigation with obstacle avoidance
  • Output — Ackermann motion driver (X-Protocol)

Network and Sensor Setup

Design intent

Reliable sensor data starts with a clean network topology. The robot runs three physical interfaces — wired Ethernet for the camera, USB for both the LiDAR and IMU, and Wi-Fi for operator communication — each kept deliberately isolated.

Mixing camera traffic and operator traffic on a shared interface is a common source of intermittent connectivity failures, especially when DHCP assigns addresses dynamically. The platform avoids this entirely: the wired port is dedicated to the camera subnet; the Wi-Fi interface is assigned a static address at the OS level, so the robot is always reachable at a known address regardless of network conditions. This makes SSH access, remote visualisation, and GUI forwarding reliable in field conditions.

Sensor data pipeline

IP camera — Delivers a continuous RTSP stream. Exposed to the middleware as a calibrated image topic pair (raw image + camera metadata). Useful for visual inspection and as a future input for vision-based localisation.

2D LiDAR — The primary navigation sensor. Multiple scan sources are merged before being forwarded to the mapping and navigation pipeline, improving angular coverage and reducing blind spots inherent to a single-unit setup.

IMU — Raw accelerometer and gyroscope data is passed through an orientation filter before entering the stack. This step is non-optional: feeding unfiltered IMU output into a navigation pipeline degrades pose estimation quality, particularly during cornering, in ways that are difficult to diagnose after the fact.

Teleoperation

The platform supports two independent control modes, both available simultaneously.

Desktop GUI — A Qt5 application running on the onboard computer, accessible via remote display forwarding. Provides direct velocity control with configurable speed limits and a live connection status indicator.

Android app — A TCP bridge that accepts joystick input from an Android device and translates it to robot commands. No ROS installation is required on the phone, which significantly lowers the barrier for operator onboarding and field use.

Both modes publish to the same motion interface, meaning the operator can switch between desktop and mobile control without any reconfiguration. The motion driver handles the Ackermann kinematic conversion in both cases.

2D Mapping

The mapping pipeline operates in asynchronous mode, meaning the map is updated continuously as new scan and odometry data arrives — without blocking the robot's motion or the operator's control stream. This is important for real-time exploration: the robot can be driven freely while the map builds in the background.

The output is an occupancy grid — a probabilistic representation of free space, occupied space, and unknown regions — stored as a portable image and metadata file pair. This format is directly loadable by the navigation stack without conversion, keeping the mapping-to-navigation handoff clean.

  • Inputs: 2D LiDAR scan data, wheel odometry
  • Output: Occupancy grid map (image + metadata), exportable for navigation

Localisation, Navigation and Obstacle Avoidance

Once a map is available, the platform switches from exploration mode to navigation mode. The full Nav2 pipeline activates.

Localisation

A particle filter localises the robot on the pre-built map by continuously comparing incoming LiDAR scans against the expected scan at each candidate pose. Over successive sensor readings, the particle distribution converges on the robot's actual position. Odometry data is fused to propagate the estimate between scan updates, maintaining continuity during brief periods of scan ambiguity (e.g. long featureless corridors).

Path planning and execution

Navigation to a goal is split across two planners operating at different time horizons:

Global planner — Computes the optimal path from the robot's current estimated pose to the goal, treating the pre-built map as ground truth. Runs at low frequency; the output is a reference trajectory.

Local planner — Executes the reference trajectory while reacting to real-time sensor data. Continuously rebuilds a local costmap around the robot by inflating detected obstacles, and computes velocity commands that keep the robot on the planned path while avoiding collisions. Runs at high frequency.

This two-planner architecture means the robot can navigate efficiently across a known space while still responding to unexpected obstacles — a person walking across the path, a door that has been closed — without replanning the entire global route.

Recovery behaviours

The navigation stack includes a behaviour tree that handles failure cases automatically. If the robot becomes stuck or the planned path becomes infeasible, it executes a recovery sequence — typically backing up, rotating in place, and re-attempting — before escalating to a full replan. This makes the system robust to the kind of transient blockages common in real indoor environments.

Ackermann constraint handling

Unlike differential-drive platforms, an Ackermann robot cannot rotate in place. The controller configuration explicitly encodes the minimum turning radius and steering geometry of the chassis. Navigation goals that would require infeasible manoeuvres are handled gracefully — the planner routes around the constraint rather than issuing commands the hardware cannot execute.
