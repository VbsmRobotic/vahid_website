MOLA LiDAR mapping: technical overview

Sections 1–3 follow Blanco-Claraco, arXiv:2407.20465 (IJRR, 2025). Sections 4–5 describe this page’s RoboSense demos and operator workflow.

1. Core concept: view-based maps

Most traditional mapping pipelines commit early to a single metric representation—such as one large fused point cloud—and discard the intermediate structure that produced it. MOLA takes the opposite approach.

MOLA treats the view-based map (also called a simple map) as the primary map object. A view-based map is a pose graph in which each node stores a time-stamped sensor observation (a keyframe) together with its SE(3) pose, rather than a pre-fused geometric product. This preserves flexibility: from one session archive you can generate multiple task-specific metric maps—dense point clouds, 2D projections, occupancy grids, or localization-optimized representations—without repeating field acquisition.

Formal definitions, figures, and large-scale experimental results: Blanco-Claraco, “A flexible framework for accurate LiDAR odometry, map manipulation, and localization,” arXiv:2407.20465 (IJRR, 2025).

2. File formats: .simplemap vs. .mm

MOLA separates the workflow into two artifacts with different roles:

Aspect .simplemap .mm (metric map)
Contents SE(3) keyframe poses + linked raw sensor observations Geometric layers: point maps, voxel grids, 2D occupancy, etc.
Role Session archive—re-run backends or generate new metric maps at any time Deployment artifact for localization, planning, or visualization
Processing state Minimally processed; retains observational richness Filtered, merged, and layered for a specific downstream task

.simplemap serializes the full mapping session—poses and attached observations—in a reprocessable form. You can change filtering, re-run different backends, or build entirely different metric products long after acquisition.

.mm is a derived geometric product, typically produced by sm2mm (simple-map-to-metric-map) with a pipeline YAML, optimized for a specific consumer (e.g. MM Viewer or a LiDAR localization node).

Principle: keep one authoritative session record (.simplemap), then generate purpose-built metric products (.mm) as needed.

3. Front-end pipeline and ICP-based pose tracking

Modular processing architecture

MOLA composes odometry and mapping front-ends from reusable, configurable processing blocks—similar in spirit to stacking layers in a deep learning framework. Typical blocks include:

  • Motion compensation (deskewing): corrects distortion in rotating multi-beam LiDAR scans caused by motion during acquisition.
  • Downsampling: reduces point density (e.g. voxel grids) to control computational cost.
  • Spatial and self-hit filtering: removes points from the platform or other unreliable returns.
  • Local map management: maintains a rolling local map used as the registration target.

Pose estimation via ICP

Pose estimation uses an ICP-style iterative registration loop:

  1. Build point-to-point or point-to-model correspondences between the current scan and the local map.
  2. Solve for the best-fit rigid transformation (rotation + translation).
  3. Iterate until convergence.

Practical enhancements include:

  • Robust loss functions: down-weight outlier correspondences in cluttered environments.
  • Motion priors: initialize using the previously estimated velocity to shrink the search basin and improve convergence.
  • Velocity–deskewing coupling (MOLA-specific): linear and angular velocity estimates feed back into deskewing so improved motion sharpens the next input scan.
  • Scan-to-map registration: align against a growing local map, not only the previous frame—stabilizing tracking when frame-to-frame geometry is weak.
  • GICP-class objectives: surface-aware alignment for informative residuals in structured environments.

4. Two experimental configurations (same site)

Both scenarios use the same MOLA stack on RoboSense ROS 2 bags from the same location; only preprocessing and fusion differ—so effects on map quality are directly observable (see figures and YouTube links above).

Scenario A — Ground-filtered, downsampled LiDAR (no IMU)

  • Above-ground features emphasized; dominant floor plane suppressed.
  • Reduced clutter improves ICP constraint quality and interpretability.
  • Lower point density reduces storage and computation.

Scenario B — Full-resolution LiDAR + IMU fusion (no ground filtering)

  • Full ground-plane returns preserved alongside structure.
  • IMU fusion improves motion compensation and robustness under fast motion.
  • Denser surfaces—visually busier maps.

Both are valid instances of the same architecture; the difference is front-end knob settings.

5. End-to-end workflow

Follow these steps in order:

Step 1 — Data acquisition

Record a ROS 2 bag with RoboSense sensor_msgs/PointCloud2. For Scenario B, include time-synchronized sensor_msgs/Imu (e.g. /imu/data).

Step 2 — Preprocessing (optional, configuration-dependent)

  • Scenario A: downsampling and ground filtering in your launch pipeline.
  • Scenario B: full-resolution data; disable ground filtering.

Step 3 — Run LiDAR odometry

Launch MOLA LiDAR odometry (e.g. mola_lidar_odometry / mola-cli with your YAML) to estimate the trajectory and accumulate keyframes.

Step 4 — Save the simple map

Output a .simplemap file—your reprocessable session archive.

Step 5 — Build a metric map (optional)

Run sm2mm with a pipeline YAML to produce a .mm for MM Viewer or a localization stack.

Step 6 — Evaluate the trajectory (optional)

Export keyframe poses to TUM format and analyze with evo:

pip3 install evo
sm-cli export-keyframes /path/to/map.simplemap --output kfs.tum
evo_traj tum kfs.tum -p --plot_mode=xy

6. Why this architecture matters

The .simplemap / .mm split keeps MOLA workflows maintainable and future-proof. By preserving the full session as a view-based archive, teams can:

  • Re-run backends with new parameters without returning to the field.
  • Generate multiple specialized metric products from one acquisition (localization map, dense visualization, 2D grid).
  • Compare preprocessing choices—as in Scenarios A and B—with the same pipeline and evaluation tools.

Separating what the robot observed from what the stack should consume is the defining design principle of the MOLA framework.
