Read in order: each step builds on the previous one—from what MOLA stores, to how the algorithm behaves, to the two demos, to the exact file flow you can reproduce.

Step 1 — What this project is

A repeatable offline workflow: ROS 2 bag recordings → MOLA LiDAR (and optional LiDAR–IMU) odometry → a keyframe simple map (.simplemap) → optional metric map (.mm) for MM Viewer, localization, or export. The same physical site is mapped twice with different preprocessing and sensors so you can see concrete trade-offs, not just a single pretty screenshot.

Step 2 — Simple map vs metric map (MOLA vocabulary)

In MOLA, these are not two names for the same file. The simple map is the session-level, keyframe-centric archive; the metric map is a derived, geometry-first product you use for navigation and visualization.

Aspect .simplemap .mm (metric map)
Data Keyframes: SE(3) poses + linked raw / rich observations (e.g. scans) Geometric layers: point maps, 2D grids, voxels, etc.
Role Archive the mapping session; re-run backends or build different metric maps later Deploy for localization, planning, obstacle checks, MM Viewer
Processing Structured but “raw” relative to final geometry products Filtered / merged / layered for a specific task
  • Simple map (.simplemap): the foundational MOLA representation—poses plus observations. Save it when odometry finishes if you may want new metric exports or offline analysis later.
  • Metric map (.mm): produced from a simple map using sm2mm (simple map → metric map), e.g. a filtered point cloud layer suitable for mola_lo-style localization or MM Viewer.

Step 3 — Algorithm fundamentals (what runs inside MOLA)

Scan-to-map alignment. Incoming scans are registered against a local map, not only against the previous frame—helpful when frame-to-frame motion is ambiguous.

GICP-style registration. The pipeline exploits local surface structure so facades and 3D texture contribute to a stable pose.

Optional ground filtering. Suppressing dominant ground returns can sharpen structure for ICP but removes explicit floor points from what you see in the cloud.

Optional LiDAR–IMU fusion. IMU supports motion priors and motion compensation; it complements geometry but does not remove the need for good overlap and tuning.

Step 4 — Two scenarios we implemented (same site)

Scenario A — Ground-filtered, downsampled LiDAR (no IMU on this path). Cleaner above-ground emphasis; less floor-dominated clutter for interpretation and registration.

Scenario B — Full-resolution LiDAR + IMU fusion; ground filtering off. Denser surfaces and full ground returns; richer but visually busier.

Compare the paired images and the two YouTube links above—same workflow class, different knobs.

Step 5 — End-to-end workflow (what to run, in order)

  1. Record or obtain a ROS 2 bag with RoboSense PointCloud2; for Scenario B, include time-synchronized Imu (e.g. /imu/data).
  2. Preprocess (optional). Scenario A: downsample + ground filter before / within your launch pipeline; Scenario B: full resolution, no ground filter.
  3. Run MOLA LiDAR odometry (e.g. mola_lidar_odometry / mola-cli with your YAML) to estimate trajectory and accumulate keyframes.
  4. Save a simple map — output *.simplemap (session archive for reprocessing).
  5. Optional — build a metric map: run sm2mm with a pipeline YAML to produce *.mm for MM Viewer or localization stacks.
  6. Optional — evaluate trajectory: export keyframe poses to TUM and plot (e.g. with evo): 
pip3 install evo
sm-cli export-keyframes /path/to/map.simplemap --output kfs.tum
evo_traj tum kfs.tum -p --plot_mode=xy

Step 6 — Why it matters

The split between .simplemap and .mm is how professional MOLA workflows stay flexible: you keep one truthful session record, then mint metric products tailored to navigation or visualization. Showing two scenarios on one site makes the impact of preprocessing and IMU tangible for anyone evaluating map quality or deploying autonomy.
